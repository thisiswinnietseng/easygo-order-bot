# EasyGo 訂購自動化 — 流程規格

> 獨立專案，與 [kkday-ticket-bot](../kkday-ticket-bot) 分開維護。
> 目的：從 KKday BE2 訂單頁抓資料，自動到供應商平台 EasyGo (Japan Holiday Travel Kyushu) 下單、支付、回填憑證。

## 輸入

- KKday 訂單編號（人工提供，例如 `26KK280740293`）

## 前置條件 / 中止條件

- 「導覽語系」是否為 `zh-tw`、「旅客備註」是否有內容，**都不再中止自動化**（2026-07-14 起改為僅記錄，仍正常送出訂單/付款）。這兩個值會抓下來，顯示在執行紀錄與 Excel 匯出中，供事後人工回顧；旅客備註內容**不會**被帶入 EasyGo 訂單（EasyGo 表單本身沒有對應欄位可填）。

## 涉及系統

| 系統 | 網址 | 帳號 |
|---|---|---|
| BE2（KKday 訂單後台） | `be2.kkday.com/order/order_view/{訂單編號}` | 既有登入 session |
| EasyGo（供應商平台） | `https://easygojp.com/#/sysHome` | ID: `KKDAYJP`（公司共用帳號，密碼寫在 `.env` 的 `EASYGO_PASSWORD`，跟主管或 Winnie 索取，語系選簡体中文） |

## 完整流程（22 步）

1. 進入 be2 訂單頁面，讀取「導覽語系」（不論是否為 `zh-tw` 都繼續，僅記錄於執行紀錄）
2. 捲動至「訂單處理進度」區塊，依序點擊「OP領取鍵」→「已訂出待供應商回覆」
3. 登入 EasyGo（切換語系為簡体中文，帳密登入）
4. 用供應商商品ID搜尋商品 → 固定填 `240624000003`
5. 點擊搜尋結果，進入商品頁面
6. 選擇日期 = be2 訂單「出發日期」
7. 類別固定選 **30%** 那一列，填入人數 → 系統顯示「結算金額(JPY)」→ 回 be2「訂單成本」頁比對「訂單總成本」，**須一致才繼續**；不一致則**中止該筆訂單並記錄原因**，不繼續作業
8. 「确认单接收邮箱」→ 固定填 `op-ib@kkday.com`
9. 「确认订单使用语言」→ 固定選 **簡体中文**
10. 「游客姓名」（姓氏/名字）← be2「旅客」頁面**主要聯絡人**（訂單旅客有多人時，只填主要聯絡人一筆）
11. 「游客手机」← be2「訂購人」頁面（國碼 + 電話）
12. 「国籍」← be2「訂購人」頁面（國家）
13. 「在日本可使用的通讯软体（仅用于紧急联络）」→ 固定填 **X**
14. 「客人郵箱」← be2「訂購人」頁面 Email
15. 「参加人员是否含有3-12岁儿童」→ 不填
16. 「是否携带大件行李」→ 不填
17. 點擊「添加备注」→ 跳出「添加备注」視窗，textarea 填入 **KKday 訂單編號**（例如 `26KK285440273`）→ 點擊「确定」（寫入「订单备注说明」，僅分銷商內部可見，方便日後對照兩邊系統）
18. 確認訂單詳情無誤 → 提交訂單
19. 直接點「立即支付」（金額已經在步驟7填人數時跟 be2「訂單總成本」比對過，付款頁不再重複比對——實測付款頁的金額欄位是不穩定的唯讀 DOM 結構，容易誤判中止）
20. 取得 EasyGo 訂單ID（例如 `12558815`）→ 回 be2「發送Voucher」分頁，填入「供應商訂單編號」→ 點擊「儲存」
21. 點擊「產生KKday憑證」→ 彈出「上傳成功」視窗 → 點擊「查看所有檔案」
22. 「發送Voucher」分頁點擊「寄送」→ 回「訂單處理進度」點擊「已寄出Voucher」→ 完成

| 确认订单使用语言 | 固定：簡体中文 |
| 游客姓名（姓/名） | be2「旅客」頁 主要聯絡人 姓氏 / 名字（多旅客只填主要聯絡人） |
| 游客手机 | be2「訂購人」頁 國碼 + 電話 |
| 国籍 | be2「訂購人」頁 國家 |

## 執行紀錄回報欄位（不影響是否送出訂單，僅供事後回顧）

| 欄位 | 來源 | 說明 |
|---|---|---|
| 導覽語系 | be2「商品」頁 | 記錄實際值（可能非 zh-tw），不中止流程 |
| 旅客備註 | be2「旅客」頁 | 記錄備註原文（可能是特殊需求），不中止流程、不帶入 EasyGo 訂單 |
| KKday商品名稱 / KKday商品編號 / KKday套餐名稱 | be2「商品」頁（欄位：商品編號(新商品)／商品名稱(zh-tw)／套餐名稱(zh-tw)） | 讓 OP 事後比對每筆訂單對應哪個商品/套餐，判斷是否需要人工去 JPH 補填資料 |
| 旅客資料規格 | be2「旅客資料規格」頁「乘客資料」區塊 | 通用讀取該區塊所有欄位，篩掉空值/預設值（0、請選擇）後留下有意義的內容（例如帶嬰兒同行），不中止流程 |

## 例外與批次處理規則

| 情境 | 規則 |
|---|---|
| 導覽語系非 zh-tw | 不中止，正常送出訂單，於執行紀錄/Excel 匯出記錄實際值供事後回顧 |
| 「旅客」頁面「旅客備註」有文字內容 | 不中止，正常送出訂單（備註不會帶入 EasyGo），於執行紀錄/Excel 匯出記錄備註內容供事後回顧 |
| EasyGo 結算金額 ≠ be2 訂單總成本 | 中止該筆訂單，記錄原因，不繼續作業 |
| EasyGo 搜尋不到商品 / 帳戶餘額不足 等其他例外 | 中止該筆訂單並記錄原因，不影響其他訂單 |
| 一次處理多筆 KKday 訂單編號（批次） | 單筆失敗只跳過該筆，繼續處理下一筆；結束後統一回報「成功/失敗清單＋失敗原因」 |

## 目前規格中仍為初版假設（尚未經使用者最終確認的細節）

- 商品固定為單一 EasyGo 商品ID `240624000003`（系島一日遊）。**多商品對照表已建立**：`product_mapping.json`，欄位為 `kkday_product_id`（KKday商品編號）/ `kkday_package_id`（套餐名稱全文，單一套餐商品可留空）/ `jph_product_id`（JPH商品編號）/ `jph_category`（JPH下單類別，如「30%」）/ `name`/`supplier`（備註用）/ `needs_beef_field`（選填，見下方牛肉餐說明）。要支援新商品時，直接在這個檔案的 `mappings` 陣列加一筆即可，不用改程式碼。`automation.py` 已有 `find_jph_mapping(kkday_product_id, kkday_package_id)` 查表函式，**但尚未接上 be2 商品編號/套餐名稱的抓取**（目前 `check_preconditions_and_extract` 還沒抓這兩個欄位），所以還是走原本單一商品的固定流程，還沒有實際用查表結果下單。

## 多商品擴充規劃（2026-07-15，已確認細節，尚未實作）

**訂單自動搜尋**：新增網頁按鈕（手動點擊觸發，非背景排程），觸發後自動到 be2 訂單查詢頁
（`https://be2.kkday.com/order/order_list`）依 `product_mapping.json` 裡每一個 `kkday_product_id`
查詢：訂單狀態=處理中、KKday訂單處理狀態=旅客資料已齊全、排序方式=出發日期排序早到晚。查詢結果列表本身
就會顯示每筆訂單的「商品編號」與「套餐名稱」，所以商品/套餐比對可以直接在列表這層做，不用逐筆點進訂單
明細頁才知道。找到符合對照表的訂單 → **直接自動下單付款，不用逐筆跟使用者確認**。查無對照表的套餐
（例如 32511 底下「福岡出發九州一日遊｜不含午餐」目前沒有對照）→ 該筆中止並記錄「找不到對照表」，
不下單，避免下錯商品。

**「是否可以食用牛肉」欄位**（`product_mapping.json` 該筆 `needs_beef_field: true` 才需要，目前
32511 兩個套餐皆需要，166716 不需要）：
- 這是 JPH confirmOrder 頁上的一個欄位（`是否可以食用牛肉：`），跟「国籍：」「在日本可使用的通讯
  软体：」「客人郵箱：」同一類——**整頁只有一個，不是每個「游客N」區塊各自一個**，用現有
  `_fill_by_label` 就能填，不用像游客姓名/手機那樣每個區塊迴圈填。
- 填入格式比照頁面本身的提示文字：`可/不可（不可的話請具體寫明幾位不食用牛肉）`。
- 判斷規則：讀該筆訂單「每一位」旅客（不只主要聯絡人）在 be2「旅客」分頁的「餐食」欄位（在旅客
  資料表格點開該位旅客、展開「更多詳細資訊」才看得到），餐食下拉選單選項為
  `猶太教餐 / 素 / 穆斯林餐 / 葷`（「請選擇餐食」為未選狀態）。規則：**葷 = 可以吃牛肉，其餘
  三種（含未選）一律視為不可以吃牛肉**。
- 全部旅客都是「葷」→ 填 `可`；有 N 位不是「葷」→ 填 `不可，N位不食用牛肉`。
- 游客區塊本身（游客姓名/游客手機）維持現行做法，只填主要聯絡人一份資料到每個區塊，不特別對應
  「哪個游客區塊是哪位真實旅客」。

## 已知限制

- **同一張 KKday 訂單不要重複執行「非 dry-run」**：目前沒有「送出前先確認 EasyGo 是否已有此訂單」的防呆機制。若對已經下單付款過的訂單再跑一次真的送出，會在 EasyGo 端產生第二筆重複訂單（重複扣款）。be2 端的狀態按鈕本身是安全可重跑的（已完成會自動跳過），風險只在 EasyGo 下單這一步。
